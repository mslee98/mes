# 제품·제품 정의 — 프론트엔드 UX 가이드

> **목적**: DB/ERD가 아니라 **사용자 흐름·화면·자동화 시점**을 기준으로 제품(`product`)과 제품 정의(`product_definition`)를 어떻게 다룰지 정리합니다.  
> 백엔드 스펙·API 세부는 별도 문서·스웨거와 맞춥니다.  
> **도메인 전체(품목·리비전·발주 라인·BOM·도면)**: [DOMAIN_PROCESS_STRUCTURE.md](./DOMAIN_PROCESS_STRUCTURE.md)

---

## 핵심 한 문장 (프론트 전달용)

**제품 목록은 계속 단순하게 유지합니다.** 사용자는 MARKOS 같은 **대표 제품**을 먼저 선택하고, 시스템은 **발주유형 / 프로젝트 / 기본 설정**에 따라 내부적으로 적용 가능한 `product_definition`을 **자동 추천**하거나 **선택**하게 합니다.

---

## 1) 왜 제품을 나누는가

| 문제 | 방향 |
|------|------|
| 제품 목록에 「MARKOS 군수 v1」「MARKOS 민수 v2」까지 모두 노출하면 선택이 과도하게 복잡해짐 | 사용자에게는 **대표 제품(MARKOS 하나)**만 보이게 한다 |
| 실제 생산·구매·BOM 기준은 버전·유형별로 달라야 함 | **상세 생산 정의**는 내부적으로 `product_definition`으로 따로 관리한다 |

즉 **목록 단순화**와 **업무 정확도**를 동시에 맞추기 위한 분리입니다.

---

## 2) 사용자는 화면에서 어떻게 느껴야 하나

원하는 인식:

- ❌ “복잡한 정의를 내가 직접 다 고르고 관리한다”
- ✅ “**제품만 골랐더니**, 시스템이 **맞는 기준(definition)**을 붙여 준다. 필요하면 바꿀 수 있다”

권장 흐름(느낌):

1. **제품 선택** (대표 제품)
2. **발주유형 선택**
3. **프로젝트 선택**
4. **시스템이 정의 자동 추천** (또는 후보만 좁혀 줌)
5. **필요 시에만** 정의 변경

---

## 3) 화면 구조 (정보 아키텍처)

### 제품 도메인

```
제품 목록
  → 제품 상세
      → 제품 정의 목록
          → 제품 정의 상세 (구성 / BOM / 공정 탭)
```

### 발주 등록

```
발주 등록
  → 거래처 선택
  → 발주유형 선택
  → 프로젝트 선택
  → 제품 선택 (대표 제품)
  → 정의 자동 추천 또는 후보 중 선택
  → 발주 라인 저장
```

---

## A. 사용자 흐름 문서 (체크리스트)

1. 사용자는 **발주유형**과 **프로젝트**를 먼저 선택한다. (정의 후보를 좁히는 입력)
2. 사용자는 **대표 제품**을 선택한다.
3. 시스템은 해당 조건으로 **product_definition 후보**를 조회한다.
4. 후보가 **1건**이면 **자동 선택**한다.
5. 후보가 **2건 이상**이면 사용자가 **선택**한다 (드롭다운 또는 모달 등).
6. **최종 선택된 definition** 기준으로 발주 라인을 생성·저장한다.

---

## B. 화면별 요구사항

### [제품 목록]

- **대표 제품만** 노출 (민·군·v1·v2를 한 목록에 섞어 쌓지 않음)

### [제품 상세]

- 제품 **기본 정보**
- **정의 목록** 탭 (해당 제품에 매핑된 `product_definition`들)

### [제품 정의 상세]

- **기본 정보**
- **구성 품목**
- **BOM 템플릿**
- **공정 템플릿**  
→ 탭 등으로 **관심사 분리** (한 화면에 전부 몰아넣지 않기)

### [발주 등록]

- **product** 선택 UI (대표 제품 중심)
- **definition** 자동 매핑 / 후보 선택 UI  
  - 후보 1개 → 자동 선택 (별도 클릭 최소화)  
  - 후보 N개 → 선택 모달 또는 드롭다운  
- 선택된 정의의 **버전·유형** 등 식별 정보를 **요약 표시** (사용자가 “무엇이 붙었는지” 인지 가능하게)

---

## C. 표시 규칙

| 구분 | 기준 |
|------|------|
| **화면 라벨·목록·카피** | **product 중심** (“MARKOS”, “이 제품”) |
| **저장·API 페이로드** | **productDefinition 중심** (`product_definition_id` 등) |

사용자에게는 **item 단위 마스터**가 아니라 **product 단위**로 인지시키고, 저장 시에만 definition id가 박히는 형태가 이상적입니다.

---

## 프론트 커뮤니케이션에 쓰기 좋은 표현

- 사용자 **선택 흐름**
- **자동 추천** / **자동 선택**
- **후보 선택**
- **표시용 product** / **내부용 definition**
- **화면 복잡도 최소화**
- **탭 분리** (정의 상세의 구성·BOM·공정)

---

## 구현 시 참고 (현재 코드베이스)

- 발주 등록 화면: `src/pages/OrderForm.tsx`
- 대표 제품: `GET /products` 등으로 조회, 셀렉트로 선택
- 정의: 제품별 `GET .../definitions` 패턴(`src/api/products.ts`의 `getProductDefinitions` 등)으로 후보 조회 가능
- **현재** 신규 등록에서는 UI에서 정의 열을 숨기고, **해당 제품의 정의 목록 중 첫 항목**을 자동 사용하는 단순화가 들어가 있을 수 있음 → 본 문서의 **발주유형·프로젝트 연동 후보 조회·다건 선택 UI**는 추후 백엔드 API와 함께 맞추며 확장하면 됩니다.
